Files & Folders

Let's start with the absolute basics: files and folders. I'm sure you all know what a file is—you've probably downloaded a picture of PDF off of the internet. All the data that you download is saved under a certain name like cat.png somewhere on your hard drive inside your computer. Your computer has lots of these "files" with collections of data scattered on your hard disk inside of your computer:

 

This is what these hard drives look like, by the way. You might recognize them if you've ever taken apart a computer or seen what it looks like on the inside. The first one is an old hard drive and the second is a newer, solid-state drive. All you really have to know is that all the files (like the pictures you download from the internet) are stored somewhere on here. Now, if you just had a bunch of these files scattered all over the palce, it would probably be pretty difficult to find where they are on this hard drive. Thankfully, we don't have to deal with this, because the people who made the operating systems we use every day (like Windows, MacOS, Linux, etc.) implemented a common concept in computing called a filesystem. A filesystem provides many features, and one of its most important features is the directory tree.

Fact: folders within other folders are often referred to as subfolders or nested folders

Folders are also referred to as directories (you might recognize this from the name "directory tree"). Try to take a look at the directory tree diagram from earlier. Can you piece together what it might signify?

The idea you should be able to take away from it is that directory structures can be visualized in an upside-down tree with many branches; larger branches split off into smaller ones with nested directories, etc. There's also what we call a "root" folder under which all folders and files are stores. As you can imagine, this whole directory tree structure makes it really easy to refer to certain files, organize them, and reference one another relative to each other. You should be familiar with the standard UNIX notation used to designate the location of a file or folder. You'll be interacting with this if you're using a Linux or Mac computer (as well as Windows if you're using WSL).

The "root" (top-most) directory under which all other directories and files are nested is referred to as / (forward slash). To refer to a folder under the root directory called "Pictures", you would use /Pictures. To refer to a file under the "Pictures" folder called "image.png", you would use /Pictures/image.png. Essentially, you would refer to a file or folder (the general term encompassing both of these is called a "path"), you would separate all of its parts with a forward slash. For example, /a/b/c/d/e.png is a valid path, pointing to the file called e.png under the directory called d, under another directory called c, under another directory called b, under another directory called a, under the root directory.

If you're using Linux or WSL, all the files for your specific user (including the files you download from the web) are located in the directory /home/your_username, whereas for MacOS, it's called /Users/your_username. If you want to actually play with these directories and browse them you can use your File Explorer or Finder (or equivalent). I'd encourage you to open up your "Downloads" folder and take a look at what's in it (just like what I did above). You should also be able to create new files or folders in this window by right-clicking on it and hitting "create file" or "create folder". It is actually possible to do all of these actions from your command line (terminal) and you'll reference files and folders using the notation we just described! Let's check it out:

Open up your terminal; you should see something like this:

You'll notice that there's ~ (tilda) character that shows up when you first open your terminal. This whole ~ $ thing you're seeing (might be different for you, but there should be a ~ and a $ somwhere in there) is called a shell prompt, and you can customize it if you want, like I have with colors and less information. In your terminal, you can type out commands and they'll show up after this prompt. Commands, more specifically "shell commands", are a language that you can use to speak to your computer through the terminal. Your terminal runs a program to interpret these commands and it's called bash. We'll be using bash a lot over the course of this post and it's a great skill to build for the rest of your career as a developer!

Some quick basics: bash understands commands in this format:

<program> [arguments...]

Where program is some program we want to run and arguments is a space-separated list of parameters we can use to modify the behavior of the program we intend to call. For example, the ls command lets you list all the files in the current "working directory". If you run ls, you'll see a list of files like this:

Yours might look a little different than mine, but you should see directories like "Downloads", "Desktop", and more. Remember the ~ character in our prompt? That meant that the "working directory" that your terminal started with was your home directory. ~ is just an abbreviation for /home/safin or /Users/safin (replaced with the corrent username), which, as we already talked about, is where all the files for your specific user are stored. We just listed all the files and directories under ~ because the default behavior of the ls command is to list all the files in the current working directory. You can also use the cd command to change your "working directory". For example, you could run cd Downloads to change to your downloads folder, run ls again, and you should see all the files in your Downloads folder. Notice how we didn't need to specify the full path /home/<user>/Downloads or /Users/<user>/Downloads because our current working directory was already our home directory and cd understands relative paths. We'll give you a little time to play with the cd and ls commands right now. Talk to your peers and navigate your filesystem with these commands!

Git

Git is a "version control system" that allows developers to track different versions of their project. It's got way more features than just that, but let's explore the basics for now.

Why does Git exist?

Developers write lots of code. Say you're a developer who's maintaining a huge codebase with hundreds of files and directories. One day, you might decide to make sweeping changes across the codebase to implement a new feature that a customer requests. If it ends up not working, you're left with a large, broken codebase, and no way to revert back to your original working code to start from scratch. Enter git: a program that allows you to track modifications to files and folders so that you can test and roll back changes easily. Git provides a command line tool called (big surprise here) git. Let's explore it by example

Git Usage

You can set up Git to track versions in any directory containing a project. Let's make a new directory somewhere under our home directory to set up Git and use it: 

xxxxxxxxxx
~$ cd vscode # or projects, etc.
~$ mkdir new-repository
~$ cd new-repository
~$ git init

Note: the mkdir command allows you to create a new directory, and the first "argument" (first space-separated parameter) you give it will be the name of the new folder.

The git init command allows you to instantiate the Git version control system in your current working directory. Essentially, we just set up Git to track versions in this current folder. You'll know if it worked if you are able to see a new .git folder pop up in your current working directory (what command can you use to check this? Hint: try the -a argument). You've just set up a Git repository, which is just a term for a folder in which Git is being used.

Note that you don't have to do this if you ran git clone to download a Git repository—more on that later.

Now, let's create a new file in our new folder with the "New File" button:

Doesn't really matter what it's called; this is just a demonstration. Once you create the file, enter some random content, and hit the save keybind on your computer. The file should be highlighted in the file explorer tab on the right in green:

This indicates that the file is a new file that is untracked by git. Remember how we talked about git being a version control system? When we ran git init to instantiate Git in our current working directory, it silently created an "initial version" of our codebase where there were no files. As such, VSCode marks the new file we made as "untracked" because Git doesn't quite know about it yet. For Git to understand that we want to "stage" this file (include it in the next "version" of our project), we can run the git add command followed by the path of the file we would like to track. Since we're using VSCode, we can navigate to the "Source Control" tab on the left and do this from the graphical user interface:

Hitting the "+" button will stage the file, getting it ready to include it in the next mini "version" of our repository.

You'll notice that the file is now marked with an A, meaning it's been added to Git in its current state. Notice that if you modify the file again, it'll be marked with an M for modified and show up like this in VSCode:

You'll notice that you still have staged changes for the new file you created (which, if you click on, you can see the difference between that staged version and the initial, blank version). However, you can also see that new-file shows up with some changes, and if you click on it, you can see the difference between the current version and the staged version:

If you stage it again with the "+" button, it will stage the file in its most updated state. Once we're ready with all the staged changes (as in, you've made a sweeping change to your repository such that it should be updated to a new "version"), we're going to use the git commit command to assign a title for this new "version" so that we can roll back to it in the future if needed.

A commit is a snapshot of all the changes in your repository, and we can easily create one with the VSCode interface. Just enter the name of the snapshot so you can refer back to it later and hit the "Commit button". You'll notice that there's no longer any highlights or symbols on the files in the file explorer tab. This is because since the last snapshot/revision/commit that was created, there have been no changes, which should make sense!

GitHub

At this point, I'm sure you've been to https://github.com before, but maybe aren't sure exactly what it's for. GitHub is a website that allows people to share their Git repositories on the internet and collaborate with other people on them. It provides a visual interface where you can browse earlier revisions of a project as well as, outside of Git, create projects and milestones to track where you want to go with your code in the future.

By convention, developers create a file called README.md in their git repository to explain what the code in the repository actually means. This file can be written in Markdown syntax, which is basically an easy way to create documents with headings, pictures, etc. If you create a README file and commit it, you should be able to "sync changes" to upload your latest revision to GitHub like so:

You can actually download repositories from GitHub and work on them, similar to what you might have done for the student repository. Git provides a command called git clone to download a repository from GitHub using its Git URL. A Git URL is basically a web link (starts with HTTPS) that refers to where the repository is stored on the internet. To find the Git URL for a project, you can hit the "Clone this repository" button here:

Then, from the command line you can run the git clone <URL> command in your vscode directory. Opening VSCode to that specific directory will allow you to manipulate that Git repository very similar to how you have been in this new test repository.
